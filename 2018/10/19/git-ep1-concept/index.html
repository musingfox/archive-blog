RE:從零開始的git 概念篇

10月 19, 2018

首先先解釋git的概念

Tree

git把code的變化以類似tree的方式呈現，每當我們要產生新的紀錄時，就會在tree上面加上新的節點(commit)，也可以在節點上開分支(branch)

示意圖：

每個節點都是一個commit，可以看到B點的時候產生了兩個分支，分支之間可以彼此切換，後面還會再提到分支之間的互動，接下來要說的是git怎麼紀錄檔案變化

States

git中把檔案的狀態分成三個states

  • modified working directory
  • staged staging area
  • commited git directory

modified代表你對這個檔案做了改動，git會比較現在的內容跟原本內容，有改動的檔案就會被視為modified，變動後的檔案我們可以將它放進staging area，檔案會變成staged的狀態，通常這代表我們確認這部分的修改沒問題，但還沒結束，我們必須將staged的檔案commit，才會放入git directory中，也就是在tree裡面加入新的節點

開發時的流程會是：

  1. 修改完檔案後儲存，git偵測到檔案變動，將檔案視為modified
  2. 將檔案放入staging area中 git add file
  3. 將staged file commit git commit

commit的時候，git會要你寫入message讓你之後方便尋找

Branch

前面提過git像是一個tree，上面有很多的節點，既然是tree，當然應該要有分支，而在git中，我們可以在節點上開新的branch，代表我們可以在同樣的基礎下開發不同版本的code或是同時開發不同的功能。

通常我們會有一個master branch，所以通常我們會在master上開分支進行開發，當開發完之後，我們可以把分支接回(merge)原本的master branch，merge之後的圖會像下面這樣

要注意的是，開發過程中可能會有改了同樣的檔案，但是內容不相同的狀況，這時候要merge就會產生衝突(conflict)，發生conflict的時候merge會無法進行，merge的人必須要先把衝突解決才能merge

Distributed

有了git的基本概念，現在可以說一下git的應用狀況

首先，git讓我們可以為自己的程式改動留下紀錄，同時開發不同的功能後結合在一起。git並不只是為了個人使用，它能夠讓開發者間可以同時開發。最重要的是，git是一套分散式的版本控制系統。

早期的版本控制系統是集中式的，集中式的特色是開發者間的檔案是同步的，也就是當某個人做改動時，其他人的檔案也會有更動，但這也代表我們必須保持在連接網路的狀態才能同步，也就是當我們寫完一部份想要commit的時候會卡住，沒辦法進行接下來的開發。

分散式的好處在於，每個人都有一份自己的檔案庫，代表你不用連網也可以發出commit，任意的修改你local端的程式，我們需要的只是一個雲端的檔案庫，大家都可以從這邊複製(clone)一份下來，然後進行開發，開發完之後我們可以push上雲端讓大家使用，也可以pull下來更新自己local的檔案庫。

分散式讓我們可以不被綁在需要聯網同步檔案庫的困境，我們只要clone之後開始開發，然後透過pull跟push共享彼此的開發成果。

總結

git讓開發者可以方便的共享檔案庫，更有效率地做開發，下一篇再來講git實際使用的方式
